# Abstract

C++ has vocabulary types and computational contexts that are individually well
understood, but it lacks a uniform way to traverse a structure while producing
contextual results and then *transpose* the result into a single outer context.
This problem appears today in combinations such as containers of `optional`s,
containers of senders, and lanewise or zipped structured computation.

This paper proposes a shape-preserving traversal facility together with
`transpose` operations that convert a structure of contextual values into a
contextual structure. The proposal is based on *independent contextual
composition* rather than on general sequential dependence.

The paper also proposes a bundled customization model. That mechanism is
included because current trait-only and per-operation customization techniques
are not strong enough to bundle applicative primitives and derived operations
coherently.

> **Status of this draft.** This is an outline skeleton accompanying the
> `beman.transpose` reference implementation. The required three-domain example
> set (`optional`-like, sender-like, SIMD/zip-like) is implemented and flows
> through one front-door API; see the `examples/` and
> `tests/beman/transpose/transpose.test.cpp` of that library.

# Before / After

The operation flips `structure<context<T>>` into `context<structure<T>>`,
preserving shape.

::: cmptable

### Before

```cpp
std::vector<std::optional<int>> in = ...;
std::optional<std::vector<int>> out{std::vector<int>{}};
for (auto& o : in) {
    if (!o) { out.reset(); break; }
    out->push_back(*o);
}
```

### After

```cpp
auto out = beman::transpose::transpose(in);
// out : std::optional<std::vector<int>>
```

---

```cpp
std::vector<zip_list<int>> in = ...;   // N lanewise lists
std::size_t lanes = in.empty() ? 0 : in.front().finite_size();
for (auto& zl : in) lanes = std::min(lanes, zl.finite_size());
zip_list<std::vector<int>> out;
out.data.resize(lanes);
for (std::size_t lane = 0; lane < lanes; ++lane)
    for (auto& zl : in)
        out.data[lane].push_back(zl.data[lane]);
```

```cpp
auto out = beman::transpose::transpose(in);
// out : zip_list<std::vector<int>>
```

:::

The same verb serves a lanewise (SIMD/zip) context and a deferred (asynchronous)
context with no per-domain wording:

```cpp
beman::transpose::transpose(vector_of_ziplists); // -> zip_list<vector<T>>
beman::transpose::transpose(vector_of_senders);  // -> sender<vector<T>> (P2300)
```

> **Status of the deferred domain.** All three domains are demonstrated end to
> end in the reference implementation. The deferred domain runs through *real*
> `std::execution` ([@P2300]) senders provided by `beman.execution`:
> `transpose(vector<S>)` yields one concrete `sender<vector<T>>` that, when run
> (e.g. via `sync_wait`), starts every child sender and gathers the results.
> Because a `std::vector` is homogeneous, the element senders share one concrete
> type and no type erasure (no `task<>`, no `std::function`) is required; the
> tests cover ordering, deferral until run, and error propagation.

# Motivation

## A chore everyone has written

Consider a `std::vector<std::optional<int>>` that you want to turn into a
`std::optional<std::vector<int>>`: a vector of values *if every element is
present*, and "no value" otherwise. The data structure is trivial and the intent
is obvious, yet there is no standard operation for it. Every codebase grows its
own loop:

```cpp
std::optional<std::vector<int>> collect(const std::vector<std::optional<int>>& in) {
    std::vector<int> out;
    out.reserve(in.size());
    for (const auto& o : in) {
        if (!o) return std::nullopt;   // early-exit, state threading
        out.push_back(*o);
    }
    return out;
}
```

This loop is easy to write and easy to get subtly wrong: the early exit, the
accumulator, the reset-on-failure, the reservation. It is also *not reusable*.
The moment the element type stops being `optional` — it is an error type, a
future, a parsed result — the loop is rewritten from scratch even though the
shape of the problem has not changed.

## The shape of the problem

The recurring shape is a structure whose elements each carry a *context*, and a
desired result that moves the context to the outside while preserving the
structure:

```
structure<context<T>>   ->   context<structure<T>>
```

We call this operation **transpose**, by direct analogy with transposing a
matrix: the two axes — *structure* and *context* — swap places, and nothing
else moves. A `vector` of three `optional`s becomes one `optional` of a
three-element `vector`. The element order and count are preserved; only the
nesting is inverted.

This is a live problem over familiar and standard-adjacent types, in at least
three independent contexts:

- **Fallible value.** `vector<optional<T>>` → `optional<vector<T>>`: succeed
  with all values, or fail as a whole. The same shape applies to
  `expected`-like and status-carrying result types.
- **Deferred result.** `vector<sender<T>>` → `sender<vector<T>>`: a structure of
  asynchronous computations becomes one asynchronous computation that produces
  the structure, with no work performed until the composed result is run. This
  is the `std::execution` ([@P2300]) shape of "gather these results."
- **Lanewise / SIMD.** `vector<zip_list<T>>` → `zip_list<vector<T>>`: a
  structure of positional (lane-indexed) computations becomes one lanewise
  computation over the structure, truncating to the common width.

The lanewise domain has a real standard analogue in `std::simd::vec<T>`
([@P1928]), and the relationship is instructive. `zip_list` *carries the
`transpose` verb* because it can hold a composite payload (a
`zip_list<vector<T>>`). `std::simd::vec<T>` cannot: it holds only vectorizable
scalars, so `vec<vector<T>>` is ill-formed and `transpose(vector<vec<T>>)` is
not expressible. `std::simd` instead demonstrates *the same independent,
position-wise composition* at the applicative-operation layer — broadcasting
with `pure` and applying a scalar function across all lanes with `invoke` /
`zip_with`. So `zip_list` proves the verb and `std::simd` proves the semantics
are literally hardware data-parallelism; the reference implementation ships both
and checks that they agree lane for lane. (See the customization section for why
`std::simd` is also the cleanest evidence that the bundled mechanism must admit
more than one primitive core.)

Three unrelated domains, one operation. Today each is served, if at all, by a
bespoke loop. None of these is reducible to the others by `transform`, `ranges`,
or `zip` — as the next section explains.

## Independent composition, not sequential chaining

The defining property of these contexts is that the per-element computations are
*independent*. Gathering a vector of senders does not feed one sender's result
into the next; the lanewise example composes position by position with no
cross-position dependence; collecting `optional`s reports overall success
without one element's value steering another's computation.

That independence is exactly what distinguishes this operation from
`and_then`/`flat_map`-style sequential chaining, where each step depends on the
value produced by the previous step. The standard library already serves
sequential dependence well (monadic `optional`, sender adaptors, coroutines).
What it does not offer is a uniform way to express *independent* contextual
composition over a structure — the applicative shape — without writing the loop
by hand for every structure/context pair.

This distinction is not academic. It is the reason `transpose` can be specified
once, with a single shape-preservation guarantee and a left-to-right effect
order, and then apply unchanged to fallible, deferred, and lanewise contexts. A
design built on sequential chaining cannot make that promise, because chaining
forces an evaluation-order dependence that the independent contexts neither need
nor want.

# Proposed operations

The front door is two free function templates. Both deduce the structure's
*traversable* instance from its type and the *applicative* context from the
element or callable result; the user names neither.

## `traverse`

```cpp
template <class F, class T>
auto traverse(F&& f, T&& structure);
```

Applies `f` to each element of `structure`, where `f` returns a value in an
applicative context, and combines the per-element effects into a single effect
holding the reconstructed structure:

```
traverse : (T -> context<U>),  structure<T>   ->   context<structure<U>>
```

Semantics, all of which are normative guarantees rather than implementation
incidentals:

- **Shape preservation.** On the success path the result structure has the same
  element order and count as the input. `traverse` reconstructs structure; it
  does not filter, reorder, or flatten.
- **Independent composition.** Element effects are combined through the
  applicative's `apply`, not through monadic sequencing. No element's *value* is
  required to compute another element's effect.
- **Left-to-right effect order.** Effects are combined in element order. This is
  observable for contexts with effects (e.g. the order subcomputations are
  registered for a deferred context) and is fixed so that the operation is
  deterministic across contexts.

## `transpose`

```cpp
template <class T>
auto transpose(T&& structure);
```

The identity special case of `traverse`: `transpose(x)` is
`traverse(identity, x)`. It flips `structure<context<T>>` into
`context<structure<T>>` when the elements are *already* in a context. This is the
verb that names the headline operation; `traverse` is the more general form that
also maps a function on the way through.

## Lifted contextual application

`traverse` is defined in terms of one applicative primitive pair —
`pure` and `apply` — from which a small derived surface (`invoke`, `map`, `ap`,
`zip_with`, ...) is generated. These are the operations a *context* author
implements or obtains; they are presented in the customization section rather
than as front-door verbs, because end users reach for `traverse`/`transpose` and
only context authors reach for `pure`/`apply`.

| Operation     | Role                | Signature sketch                                  |
| ------------- | ------------------- | ------------------------------------------------- |
| `traverse`    | front-door verb     | `(T -> C<U>), structure<T> -> C<structure<U>>`    |
| `transpose`   | front-door verb     | `structure<C<T>> -> C<structure<T>>`              |
| `pure`        | context primitive   | `T -> C<T>`                                        |
| `apply`       | context primitive   | `C<(T -> U)>, C<T> -> C<U>`                        |
| `invoke`/`map`| derived             | lift a plain callable over one or more `C` values |

# Why a bundled customization mechanism

`traverse` is not one operation over one type. It is a small *family* —
`pure`, `apply`/`invoke`, the derived `map` / `zip_with` / `lift`, and
`traverse`/`transpose` on top — that must stay mutually coherent for each
context. The mechanism that customizes it has to bundle that family, not expose
it one customization point at a time.

## Why trait-only and one-CPO-per-operation designs are not enough

A trait-only design (specialize a class template of static members) gives
values and type computations but not a coherent set of *operations* with
derived defaults. A one-CPO-per-operation design (an ADL customization point per
verb) scatters the family across independent extension points with no shared
identity: nothing ties a type's `pure` to its `apply`, nothing supplies the
derived `map`/`zip_with` once the primitives are given, and a type can satisfy
some points and silently miss others. For an applicative the laws relate the
operations to each other; splitting them into unrelated CPOs makes the coherent
unit unrepresentable and pushes the derived surface into every call site.

## Bundled customization objects with a primitive/derived split

This proposal customizes a context by registering one *object* that bundles the
whole family: the author supplies a small primitive core and inherits a derived
surface. Concretely, an `Applicative` instance provides `pure` plus one
composition primitive, and the bundle derives `invoke`, `map`, `zip_with`,
`lift`, and the rest. The same shape gives `Traversable` its derived
`transpose`/`for_each` from a single `traverse` primitive. Registration is a
variable-template specialization (`applicative_typeclass<C>`,
`traversable_typeclass<S>`); it touches neither the adapted type nor a central
registry, so `std::optional` participates without being reopened.

## Alternate primitive cores are load-bearing, not decorative

A single fixed primitive does not fit every context, so the bundle admits more
than one primitive core and derives the rest from whichever is supplied — the
same way a foldable may be defined by `fold_map` *or* `fold_right`. For
applicatives the two cores are:

- **`pure` + `apply`** — the classic form (`apply : C<(T->U)>, C<T> -> C<U>`),
  used by the `optional`, sender, and `zip_list` contexts.
- **`pure` + `invoke`** — apply a *plain* function across in-context arguments
  (`invoke : (T...->U), C<T>... -> C<U>`), with no `C<callable>` ever formed.

`std::simd::vec<T>` makes this concrete and not hypothetical. It has no
expressible `apply`, because `vec<callable>` is ill-formed — `vec` holds only
vectorizable scalars. Its only expressible primitive is `invoke` (apply a scalar
function lanewise). A mechanism locked to a single `apply` primitive cannot
admit `std::simd` at all; the bundled object, by accepting `invoke` as an
alternate core, admits it with a four-line instance and no change to the front
door. This is the strongest available evidence that the customization boundary —
bundle the family, allow alternate primitive cores — is the right one, rather
than a design tuned around the contexts that happen to support `apply`.

## Three lookup modes

The registered object is a value, usable three ways: implicit
(`applicative_typeclass<C>` looked up by type), explicit (pass a specific
instance to override policy at a call site), and NTTP-pinned (bind the instance
as a template parameter so generic code resolves it once at instantiation,
independent of ADL or call-site namespace). All three are implemented and
tested.

# Design decisions and review control

To keep review focused, each major decision is classified so that consequences
of the primary goal are not relitigated as if they were free parameters. The
four classes are: **(C) direct consequence** of the primary design goal;
**(T) motivated trade-off**; **(A) arbitrary but necessary** (a choice had to be
made and little rides on it); **(D) intentionally deferred** to a companion
paper or a later revision.

## Front-door verbs and naming — class C/A

**Decision.** The public verbs are `traverse` and `transpose`; the algebraic
names (functor, applicative, traversable, monoid) appear only as
specification/rationale vocabulary, not in the front-door API.
**Alternatives.** `sequence` (Haskell's name) for `transpose`; leading with the
algebraic hierarchy. **Why R0.** "Sequence" already carries container-noun
baggage in C++, and the operation literally swaps two axes, so `transpose` is
the more honest C++ name (consequence of the operation's meaning, class C). The
exact spelling `traverse`/`transpose` is otherwise arbitrary but necessary
(class A) and open to LEWG bikeshedding.

## Independent composition rather than monadic sequencing — class C

**Decision.** The operation is applicative (independent), not monadic
(sequential). **Alternatives.** A monadic `and_then`-style formulation.
**Why R0.** Independence is the property that lets one specification serve
fallible, deferred, and lanewise contexts unchanged; a sequencing formulation
would impose an evaluation-order dependence those contexts do not want. This is
a direct consequence of the primary goal, not a tunable knob.

## Bundled customization with alternate primitive cores — class C/T

**Decision.** Customize via one registered object bundling the family, with a
small primitive core (`pure` + `apply`, or `pure` + `invoke`) and derived
defaults. **Alternatives.** Traits-only; one CPO per operation; a single fixed
primitive. **Why R0.** Applicative coherence and the `std::simd` `invoke`-core
case show a single-primitive or scattered-CPO design cannot hold the family
together or even admit some real contexts (see the customization section). The
*mechanism* belongs in this paper (trade-off, class T): the problem cannot be
credibly solved without it, so reviewers should evaluate them together.

## Left-to-right effect order — class A

**Decision.** Effects combine in element order. **Alternatives.** Unspecified
order; right-to-left. **Why R0.** Some order must be fixed for determinism
across contexts; left-to-right matches reading order and existing fold
conventions. Arbitrary but necessary.

## What is normative vs demonstrative — class C/D

**Decision.** R0 proposes the verbs and the customization boundary. Only
`std::optional` ships a registered instance; the `zip_list`, `std::simd`, and
`std::execution` contexts are non-normative demonstrations. **Why R0.** All
three motivating domains are now proven in running code: the fallible domain on
`std::optional`, the lanewise domain both as a model (`zip_list`) and on real
hardware (`std::simd`), and the deferred domain on real `beman.execution`
senders. None is proposed as a standard instance in R0; which standard contexts
should ship registered instances is intentionally deferred (class D). The async
realization also shows the *structure* side is a customization point: a vector
of senders routes to a bespoke homogeneous gather rather than the generic fold,
chosen so the runtime-sized transpose stays a single concrete sender type with
no type erasure.

# Likely objections

- **"Why not just a helper algorithm over existing containers?"** A single
  helper would hard-code one context. The contribution is the context-agnostic
  boundary: one specification that already serves three unrelated contexts, plus
  the customization that lets a fourth join without touching the front door.
- **"Isn't this `transform`, `ranges`, or `zip` renamed?"** No. Those map or
  combine values; none moves a *context* from inside a structure to the outside
  while preserving shape, and none is parameterized over the context's
  applicative.
- **"Why applicative structure instead of ordinary sequential chaining?"**
  Because the composition is independent. Chaining imposes an order dependence
  the deferred and lanewise contexts neither need nor want, and would prevent the
  single shared specification.
- **"Too abstract for the standard library?"** The front door is two verbs over
  familiar types; the algebra is rationale, not the teaching surface. The
  motivating cases are `std::optional`, `std::execution`, and `std::simd`.
- **"Why is the customization mechanism in the same paper?"** Because the
  primitive and derived operations must stay coherent, and no weaker
  customization style achieves that — the mechanism exists because the problem
  requires it, which is exactly what the merged framing forces reviewers to
  judge.

# Companion papers and scope boundary

This paper is the anchor of a coordinated set and is designed to be reviewable on
its own.

- **What this paper covers alone.** The transposition problem, the `traverse`/
  `transpose` verbs, and the bundled customization mechanism with alternate
  primitive cores — complete over `std::optional` today, with `std::execution`
  and `std::simd` as motivating contexts.
- **What companion papers add if adopted.** A *persistent measured sequence*
  paper (P-number TBD) supplies a structure with structural sharing,
  concatenation, and measured split/search that is a natural `Traversable`. A
  *recursive tree algorithms* paper (P-number TBD) supplies recursive fold/build
  verbs over variant-based trees, reusing this customization model as evidence
  that internal iteration generalizes beyond flat sequences.
- **What stays valid if companions stall.** Everything here. There is no
  normative dependence on the companion papers; they are related work and
  additional `Traversable`/`Foldable` instances, not prerequisites.

# Non-goals for R0

- Standardizing the full algebraic hierarchy.
- Solving every naming question for adjacent abstraction families.
- Turning the paper into a generic customization-framework survey.
- Making tree containers carry the motivation for the paper.
- Mandating which standard contexts ship registered instances beyond
  `std::optional`.
