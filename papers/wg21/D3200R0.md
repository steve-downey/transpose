# Abstract

C++ has vocabulary types and computational contexts that are individually well
understood, but it lacks a uniform way to traverse a structure while producing
contextual results and then *transpose* the result into a single outer context.
This problem appears today in combinations such as containers of `optional`s,
containers of senders, and lanewise or zipped structured computation.

This paper proposes a shape-preserving traversal facility together with
`transpose` operations that convert a structure of contextual values into a
contextual structure. The proposal is based on *independent contextual
composition* rather than on general sequential dependence.

The paper also proposes a bundled customization model. That mechanism is
included because current trait-only and per-operation customization techniques
are not strong enough to bundle applicative primitives and derived operations
coherently.

# Before / After

The operation flips `structure<context<T>>` into `context<structure<T>>`,
preserving shape.

::: cmptable

### Before

```cpp
std::vector<std::optional<int>> in = ...;
std::optional<std::vector<int>> out{std::vector<int>{}};
for (auto& o : in) {
    if (!o) { out.reset(); break; }
    out->push_back(*o);
}
```

### After

```cpp
auto out = beman::transpose::transpose(in);
// out : std::optional<std::vector<int>>
```

:::

The same verbs serve a deferred (sender-like) context and a lanewise (SIMD/zip)
context with no per-domain wording:

```cpp
beman::transpose::transpose(vector_of_senders);  // -> sender<vector<T>>
beman::transpose::transpose(vector_of_ziplists); // -> zip_list<vector<T>>
```

# Proposed operations

- `traverse(f, structure)` — apply an effect-producing `f` to each element and
  transpose the effects, preserving shape.
- `transpose(structure)` — `traverse` with the identity function.
- lifted contextual application: `invoke` — a plain function over N in-context
  arguments, the n-ary core, McBride & Paterson's canonical form
  `pure f ⊛ u1 ⊛ … ⊛ un`. It is the *only* application form proposed; see
  [Why not `apply`](#why-not-apply).

# Why not `apply`

The classic applicative-application spelling — `apply(cf, cx)`, a callable
held *inside* the context, applied to one argument-in-context at a time — is
deliberately not proposed, in any form (`apply`, `ap`, or currying helpers).

That formulation is an artifact of Haskell's currying: `f <$> u1 <*> u2 <*> u3`
works there because partial application re-inserts a function into the functor
at every step. C++ is not curried; reproducing the chain requires manufacturing
curried closure objects that exist only to be immediately consumed, and the
result teaches nothing — it is machinery in service of a foreign idiom.

The decisive argument is in this paper's own motivation: `std::simd::vec`
holds only vectorizable scalars, so `vec<callable>` **is not a type** and
`apply` cannot be spelled for it even in principle — while the n-ary `invoke`
(the lane-wise generator) is exactly what the hardware operation is. An
`apply`-shaped core would lock out one of the three motivating domains;
`invoke` serves all three. `apply` was never the primitive; it was a curried
serialization of it.

Contexts that *can* hold callables lose nothing: `pure(f)` is still a fine
contextual value, and applying it is `invoke(call, cf, cx)` with a plain
two-argument evaluation function. The reference implementation's law suite is
stated entirely in `invoke` form (identity, homomorphism generalized to the
n-ary core, interchange as floating a `pure` operand out of an application,
and functor composition), and it pins the *absence* of the apply forms on
every instance.

# Why not Monad

Monad is deferred, not rejected — absent because nothing proposed needs it,
not because it is unwanted.

Everything about the typeclass-object design points toward a Monad instance
eventually. A consistent generic name for sequential composition is exactly
the kind of thing the mechanism exists to provide: today's spellings —
`optional::and_then`/`or_else`/`transform` ([@P0798R8], C++23), the same
family on `expected` ([@P2505R5]), `let_value` on senders — are per-type
member functions, so no algorithm can be written once, generically, over
anything sequentially composable. That is the gap typeclasses close, and the
reference implementation's `monad.hpp` — `bind`, `join`, Kleisli
composition, an `optional` instance, `invoke` synthesized from `bind` +
`pure` — demonstrates the mechanism carries Monad without strain.

It is omitted from this paper for one reason: no operation proposed here
calls `bind`. `traverse` and `transpose` require Applicative — independent
contextual composition — and the three motivating domains work precisely
because no operand's context depends on another operand's value. Proposing
Monad in R0 would attach the family's largest naming-and-teaching
discussion to a paper whose operations never use it.

When a proposal needs to be generic over sequential composition, Monad is
the paper to write; the coordinated set leaves that slot open, and the
evidence that it slots in cleanly is already in the tree. For now.

# Non-goals for R0

- Standardizing the full algebraic hierarchy.
- A Monad abstraction, in this paper or the coordinated set (see
  [Why not Monad](#why-not-monad)).
- The classic `apply`/`ap` application forms (see
  [Why not `apply`](#why-not-apply)).
- A Foldable abstraction: the fold family's motivation is structures that
  are not ranges, and it is proposed by the recursive-tree-algorithms
  companion paper. The reference implementation's `fold.hpp` and
  `monad.hpp` are evidence that the mechanism scales, not proposed wording.
- Solving every naming question for adjacent abstraction families.
- Turning the paper into a generic customization-framework survey.
- Making tree containers carry the motivation for the paper.
