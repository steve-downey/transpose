# Abstract

C++ has vocabulary types and computational contexts that are individually well
understood, but it lacks a uniform way to traverse a structure while producing
contextual results and then *transpose* the result into a single outer context.
This problem appears today in combinations such as containers of `optional`s,
containers of senders, and lanewise or zipped structured computation.

This paper proposes a shape-preserving traversal facility together with
`transpose` operations that convert a structure of contextual values into a
contextual structure. The proposal is based on *independent contextual
composition* rather than on general sequential dependence.

The paper also proposes a bundled customization model. That mechanism is
included because current trait-only and per-operation customization techniques
are not strong enough to bundle applicative primitives and derived operations
coherently.

# Before / After

The operation flips `structure<context<T>>` into `context<structure<T>>`,
preserving shape.

::: cmptable

### Before

```cpp
std::vector<std::optional<int>> in = ...;
std::optional<std::vector<int>> out{std::vector<int>{}};
for (auto& o : in) {
    if (!o) { out.reset(); break; }
    out->push_back(*o);
}
```

### After

```cpp
auto out = beman::transpose::transpose(in);
// out : std::optional<std::vector<int>>
```

:::

The same verbs serve a deferred (sender-like) context and a lanewise (SIMD/zip)
context with no per-domain wording:

```cpp
beman::transpose::transpose(vector_of_senders);  // -> sender<vector<T>>
beman::transpose::transpose(vector_of_ziplists); // -> zip_list<vector<T>>
```

# Proposed operations

- `traverse(f, structure)` — apply an effect-producing `f` to each element and
  transpose the effects, preserving shape.
- `transpose(structure)` — `traverse` with the identity function.
- lifted contextual application: `invoke` (a plain function over N in-context
  arguments — the n-ary core, McBride & Paterson's canonical form
  `pure f ⊛ u1 ⊛ … ⊛ un`) with `apply` as the classic alternate spelling,
  derivable in either direction (GHC: `MINIMAL pure, ((<*>) | liftA2)` with
  `(<*>) = liftA2 id`).

# Why a bundled customization mechanism

[Why trait-only and one-CPO-per-operation customization do not hold the
applicative family together; the primitive/derived split, including the
Applicative dual core `pure` + (`invoke` | `apply`) — the base derives
whichever the instance does not supply, and `std::simd::vec` (which cannot
form a `vec<callable>`, so `apply` is unspellable) participates through
`pure` + `invoke` alone, the concrete case a single global `apply` hook
would lock out; overrideable defaults; the three lookup modes.]
